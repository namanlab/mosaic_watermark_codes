# MOSAIC: Set-Based Multi-Bit Watermarking for Language Models

MOSAIC is a multi-bit watermark for language-model text. It embeds a recoverable
`k`-bit payload (a user id, session token, timestamp, ...) into generated text so
that a party holding a secret key can read it back exactly, using only the
tokenizer and the key at detection time (no model access).

Every prior multi-bit text watermark reads the token stream as an **ordered
sequence**. MOSAIC treats it as an **unordered bag of parity samples**: each token,
keyed by an order-canonical hash of its local context, emits one parity check over
a key-chosen subset of the payload bits (a fountain / LT code generated by the text
itself), and the payload is recovered from the whole unordered collection by belief
propagation. That single change of viewpoint is what makes recovery robust to
reordering, insertion, and deletion by construction, and what removes the `β^k`
exact-match ceiling that limits independent per-bit decoders.

This repository contains the MOSAIC implementation, the full experiment harness,
and faithful reimplementations of six multi-bit baselines evaluated under a single
shared protocol.

```
   payload m (k bits)
        │  each generated token, keyed by an ORDER-CANONICAL hash of its
        │  local context (the SORTED previous-token window), commits to:
        ▼
   a random PARITY CHECK over the payload bits:  m[i] ⊕ m[j] ⊕ m[l]   (a fountain symbol)
        │  scramble by a keyed chip bit r, then bias the token toward
        ▼  green if the scrambled parity = 1, else red; sample as usual
   TEXT ──► detector recomputes every token's check from its context,
            reads its colour → a soft observation (LLR) of that parity,
            collects ALL of them (order irrelevant), and runs
            BELIEF PROPAGATION over the Tanner graph → recovers m + a p-value
```

`results/` is created at run time and holds the machine-readable JSON outputs
(one file per experiment stage / method).

### The MOSAIC source (`src/`)

- **`mosaic.py`** — `MosaicWatermark`, the top-level object. `generate()` embeds a
  payload during sampling (or produces an unmarked baseline with `watermark=False`);
  `detect_ids()` / `detect_text()` recover the payload and a p-value with no model;
  `perplexity()` scores text; `reconfigure()` changes payload size / bias / window
  in place so one loaded model serves many settings.
- **`content_prf.py`** — `ContentPRF`, the order-canonical keyed PRF. Every
  per-token quantity (green mask `G_t`, chip `r_t`, check subset `D_t`) is keyed on
  the **sorted multiset** of the preceding `w` tokens, never on position. This is
  what makes the collected checks a set and gives reordering invariance.
- **`fountain_bp.py`** — the order-agnostic coding core: `robust_soliton()` for the
  check-degree distribution and `bp_decode()`, a belief-propagation decoder over the
  bipartite (payload-bit ↔ check) Tanner graph. Pure NumPy, no model dependency.
- **`conv_code.py`** — a legacy rate-1/2 convolutional code with Viterbi decoding
  from an earlier (sequential) design. It is retained for reference and is **not
  used** by MOSAIC, which uses the order-free fountain code in `fountain_bp.py`.

## Installation

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Requirements: `torch`, `transformers`, `accelerate`, `numpy`, `pyarrow`,
`matplotlib`, `sentencepiece`, `protobuf`, `mauve-text`. A CUDA GPU is recommended
for Llama-2-7B (about 16 GB VRAM in fp16); GPT-2 runs on CPU or Apple MPS.

Gated models require a Hugging Face token with access to
`meta-llama/Llama-2-7b-hf` and (for the perplexity oracle)
`meta-llama/Llama-3.1-8B-Instruct`:

```bash
export HF_TOKEN=hf_xxx        # or: huggingface-cli login
```

## Data

`data/opengen_data.parquet` is the OpenGen prompt set (from the DIPPER paper,
Krishna et al. 2023). It has a `prefix` column (the prompt) and a `targets`
column (the human continuation, used as the MAUVE reference). Prompts are
truncated to 128 tokens with the model's own tokenizer before generation.

## Quick start (the MOSAIC API)

```python
from src.mosaic import MosaicWatermark

# embed a 32-bit payload
wm  = MosaicWatermark(model_name="gpt2", secret_key="key", k_bits=32,
                      delta=6.0, gamma=0.5, temperature=0.8, top_p=0.95, window=2)
msg = [1,0,1,1,0,0,1,0]*4
res = wm.generate("The future of renewable energy", msg, max_new_tokens=200)

# detect (model-free: needs only tokenizer + key)
det = wm.detect_text(res["generated_text"], prompt="The future of renewable energy")
print(det["message"] == msg, det["p_value"])

# detect straight from token ids
det = wm.detect_ids(res["generated_ids"], prompt_ids=res["prompt_ids"])

# perplexity of watermarked vs unmarked text (quality)
plain = wm.generate("The future of renewable energy", msg, watermark=False)
print(wm.perplexity("The future of renewable energy", plain["generated_ids"]))
print(wm.perplexity("The future of renewable energy", res["generated_ids"]))

# reuse one loaded model across payload sizes / bias values (no reload)
wm.reconfigure(k_bits=48, delta=5.0)
```

Key parameters: `delta` (green-list logit bias, default 6), `gamma` (green
fraction, 0.5), `temperature` / `top_p` (nucleus sampling, 0.8 / 0.95), `window`
(context window `w`, 2), `soft_mag` (detector LLR magnitude `A`).

## Reproducing the experiments

### MOSAIC (`run_experiments_full.py`)

The harness is split into four stages selected by `--experiment`; each loads at
most one heavy model at a time and writes/merges JSON into `results/`. Every stage
checkpoints and auto-resumes from its own output.

| `--experiment` | what it does | model(s) |
|---|---|---|
| `generate` | prompts + watermark; saves ids, marked/plain text, per-prompt match | generation model |
| `quality` / `quality-perplexity` | base-model perplexity (Llama-3.1 oracle) + MAUVE | oracle + gpt2-large |
| `robust-substitution` / `robust-insertion` / `robust-deletion` / `robust-reordering` | one token-level attack each (model-free) | none |
| `robust-paraphrase` | DIPPER paraphrase attack (self-generates paraphrases, then detects) | DIPPER |
| `robust` | runs all attacks; results merge into one file | DIPPER |
| `sweep-delta` | match rate + perplexity vs the bias delta | gen + oracle |
| `sweep-tokens` | bits x tokens match-rate heatmap | gen model |
| `sweep` | runs both sweeps | gen + oracle |

```bash
# full run on Llama-2-7B (stage 1 first; the rest depend on it)
nohup python run_experiments_full.py --experiment generate --model meta-llama/Llama-2-7b-hf > gen.log 2>&1 &
python run_experiments_full.py --experiment quality --model meta-llama/Llama-2-7b-hf
python run_experiments_full.py --experiment robust  --model meta-llama/Llama-2-7b-hf
python run_experiments_full.py --experiment sweep   --model meta-llama/Llama-2-7b-hf

# fast smoke test on GPT-2 (3 prompts, small oracle/paraphraser)
MOSAIC_QUICK=1 python run_experiments_full.py --experiment generate --model gpt2
```

Environment knobs: `MAIN_MAX_PROMPTS` (0 = all prompts, else the first N in
order), `SUBSET_FRAC` (fraction used for robustness/sweeps, default 0.20),
`ORACLE_MODEL`, `PARAPHRASE_MODEL`, `PARAPHRASE_TOK`, `QUANTIZE_8BIT`,
`CHECKPOINT_EVERY`, `LOG_EVERY`, and `MOSAIC_QUICK=1` for the smoke test.

Fixed protocol: 200 generated tokens, delta = 6, gamma = 0.5, window = 2, nucleus
sampling (tau = 0.8, top-p = 0.95), payloads 16 / 20 / 24 / 32 / 48 bits, seed 42.

### Baselines (`run_baseline_experiments.py`)

Runs the six reimplemented baselines under the exact same protocol (same prompts,
model, sampling, bias, payload sizes, and per-prompt messages), with model-free
detection, so the numbers are directly comparable to MOSAIC.

```bash
python run_baseline_experiments.py --method all  --model meta-llama/Llama-2-7b-hf
python run_baseline_experiments.py --method qu   --model meta-llama/Llama-2-7b-hf
MOSAIC_QUICK=1 python run_baseline_experiments.py --method all --model gpt2   # smoke test
```

`--method` is one of `mpac`, `qu`, `cyclic`, `wang`, `cohen`, `janas`, or `all`.
Same `MAIN_MAX_PROMPTS`, `CHECKPOINT_EVERY`, `LOG_EVERY`, and `MOSAIC_QUICK` knobs.

## The six baselines

All are reimplemented on the shared green-list physical layer
(`src_baselines/common.py`) and differ only in how a payload is written onto it and
read back. "Extraction" is the decode cost as a function of payload size `k`.

| Method (`--method`) | Paper | Mechanism | Extraction | Payload sizes run |
|---|---|---|---|---|
| `mpac` | Yoo et al., NAACL 2024 | position allocation of bits, majority vote per bit | O(T) | 16-48 |
| `qu` | Qu et al., USENIX Security 2025 | pseudo-random segment assignment + Reed-Solomon (GF(2^4), default [14,8]); balanced half-vocab green list per symbol value; argmax per segment then RS-decode | O(T + n·2^m) | 16-48 |
| `cyclic` | Fernandez et al., WIFS 2023 | message shifts a per-message green list; score all candidate messages | O(T·2^k) | 16-24 |
| `wang` | Wang et al., ICLR 2024 | whole message hashes the vocabulary partition; score all candidate messages | O(T·2^k) | 16-24 |
| `cohen` | Cohen et al., IEEE S&P 2025 | two keyed green lists per bit; decode each bit by which key carries the mark | O(T) | 16-48 |
| `janas` | Janas et al., 2025 | payload = a line over GF(2^{k/2}); embed points, recover by Maximum Collinear Points | O(N^2) | 16-48 |

Two implementation notes, kept faithful and documented in the source:

- **Qu et al.** Reed-Solomon error correction (`reed_solomon.py`, a GF(2^4) codec)
  and a *balanced half-vocabulary* green list per symbol value are the two pieces
  that keep the match rate high at large payloads; without them the scheme collapses
  on the green-list layer. GF(2^4) caps the codeword at 15 symbols, so at 48 bits
  (12 data symbols) only a [15,12] code is available (corrects one error), which is
  the source of its lower 48-bit accuracy.
- **Cohen et al.** is defined on an undetectable, distortion-free zero-bit
  primitive; ported to the shared green-list layer at delta = 6, its per-bit
  detector would be diluted by the tokens that do not carry a given bit, so we
  realize the same two-keys-per-bit mechanism with a key-derived block schedule and
  per-bit detection. This adaptation affects only the Cohen baseline.

`cyclic` and `wang` decode by scanning all `2^k` candidate messages, which is
exponential in the payload; they are run up to 24 bits, where the search is still
practical.
